## 3. 闭环流程

### Observe

收集输入数据哈希、编码约定、数学输出、统计 seed/试验数、历史引用、强断言、页面类别和已发布版本。

### Orient

把内容拆为数学恒等、统计描述、历史主张、语义解释、跨学科类比和艺术表达；每类采用不同证据门槛。

### Decide

按“数学错误 > 公开误导 > 历史无来源 > 版本漂移 > 体验”排序。每轮固定一个主张、反例和验收脚本。

### Act

先写失败断言；核心数据单一来源生成多载体；探索页不共享论文结论语气；修订保留勘误。

### Verify

- 两套独立循环实现；
- 精确基线与带 seed 的模拟；
- 编码方向敏感性；
- 历史引用和新颖性检索日志；
- 禁用词/内容类别 guardrail；
- 链接、JS、浏览器和 DOI/Pages 版本；
- 前提审计完整性。

### Learn

把外部纠错按数学、史学、统计、叙事分类；每次错误新增回归或内容门禁，并记录哪些旧公开版本受影响。

## 5. 质量门禁

- 64 编码唯一、循环型、固定点、阶和汉明分布断言通过。
- 随机模型、精确期望、p 值方向和多重比较透明。
- 历史主张逐条引用，传统归属不写成事实。
- “证明、非随机、因果、同构”必须达到相应证据门槛。
- GitHub Pages、论文和 DOI 版本一致或显式声明差异。
- 所有公开复现命令在干净环境运行。

## 6. 反馈指标

- 数学回归、历史引用覆盖、越界措辞和版本漂移数量。
- 独立复现成功率、外部纠错处理时间。
- 探索页被误读为事实的反馈数。
- Pages/DOI 发布门禁通过率。

## 前提审计

1. **前提审查：** 闭环假设项目愿意把出版严谨性置于叙事冲击力之上；数学复算已验证，历史和新颖性仍需外部来源审校。
2. **术语审查：** “随机基线”必须指定随机模型；“显著”必须指定检验；“文王序/伏羲序”不能自动携带作者和年代结论。
3. **因果审查：** Verify 只证明计算与声明一致，不证明历史生成机制或跨领域因果。
4. **反面审查：** 即使全部门禁通过，仍可能存在未检索文献、替代编码、后验选择和史学争议；因此结论必须保持条件化。
